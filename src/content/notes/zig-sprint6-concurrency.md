Notes from Sprint 6: a mutex counter and an atomic one, a producer-consumer queue on condition variables, and a worker pool driving a parallel checksum pipeline (the [long post](/series/zig-learning/threads-mutexes-and-a-worker-pool-in-zig) walks the code). Three things that only clicked after writing them. Note up front: in Zig 0.16 the primitives go through the `io` capability — `std.Io.Mutex`, `std.Io.Condition`, `mutex.lockUncancelable(io)`.

## A mutex guards a block; an atomic guards one word
Both counters fix the same bug — `value += 1` is really load-add-store, and unprotected it loses increments when threads interleave. But they fix it at different grains. The mutex makes an arbitrary *region* indivisible, so reach for it when the invariant spans several ops or fields. `std.atomic.Value(u64).fetchAdd` makes the *single increment* indivisible in hardware — lock-free and faster, but it can only ever protect that one update with nothing else riding alongside. And the ordering matters: `.monotonic` (the weakest) is correct for a standalone counter *because no other memory's visibility depends on the count*. The instant another thread had to read the value and then assume some other data was ready, `.monotonic` would be too weak and you'd need a stronger ordering. Pick the atomic when it's one word and one op; pick the mutex the moment it's more than that.

## The condition-variable discipline: while-loop + broadcast
Two rules, both traps. **Wait in a `while`, never an `if`.** `cond.waitUncancelable(io, &mutex)` atomically releases the lock and sleeps (so someone else can change the predicate), then re-takes it on wake — but a wake is only a *hint*, so you must re-check the condition: spurious wakeups happen, and even a real signal can be stale if another thread raced in first. `while (count == 0 and !closed) wait(...)` treats the predicate as the truth and the signal as advisory. **`close` must `broadcast`, not `signal`.** Ending a queue means waking *every* blocked consumer so its thread can exit; `signal` wakes one and the rest sleep forever. The no-jobs pool test is built to catch exactly this — start 4 idle workers, submit nothing, shut down; with `signal` it hangs, and reaching the end proves all four woke. A closed queue drains its remaining items first, then `pop` returns null.

## A join doesn't just wait — it publishes (the important one)
Clean shutdown is *close-then-join*: `queue.close(io)` wakes the workers (broadcast), they drain-then-exit, and `for (threads) |t| t.join()` blocks until each has truly finished. That a pool's `shutdown` *returns at all* instead of hanging is the proof the workers exited. But the deeper payoff is in the parallel checksum pipeline, which reads its results with **no lock**: each job writes to its own disjoint `SumResult` slot (indexed by collection order), so there's no shared mutable state — and a thread's writes *happen-before* the join that observes its exit. So once `shutdown` returns, every slot is fully written and visible, and the main thread reads them all back lock-free. The mental upgrade: **a `join` is a synchronization point, not merely a wait — it publishes everything the joined thread did.** Disjoint slots plus one join beat a mutex-guarded result map. Two more details that fall out of the same shutdown story: the pool keeps only the *first* error and per-file errors live in their own slot (so a bad file reports without aborting the batch — graceful, not abort-on-first-error); and on the startup error path you must `close` *before* you `join` the already-spawned workers, because they're blocked in `pop` and would otherwise never return — the [errdefer ladder](/notes/zig-sprint5-systems) again, but the resource being unwound is running threads.
